# Автоматизированная система обработки корпоративной почты

Проект для хакатона по технологиям программирования. Система читает файлы из папки входящей почты, определяет смысловую категорию письма, раскладывает файлы по папкам и формирует отчёт обработки.

## В чём задача

IT-поддержка получает много писем: аварии, запросы доступа, проблемы с программами, спам, финансовые документы, HR-заявки и служебные уведомления. Нужно автоматизировать первичную сортировку, чтобы критичные обращения не терялись среди обычных сообщений.

Решение должно:

- читать письма из файловой системы;
- классифицировать их по понятным категориям;
- переносить или копировать файлы в папки категорий;
- не падать на пустых, бинарных, повреждённых и неизвестных файлах;
- сохранять лог и статистику;
- иметь bash-скрипт запуска, тесты, README и `.gitignore`.

## Категории

Классификатор использует объяснимые правила с весами ключевых слов. Итоговая категория выбирается по максимальному баллу.

- `critical_incidents` — критические сбои, недоступность сервисов, ошибка 500.
- `monitoring_alerts` — автоматические уведомления мониторинга, healthcheck, disk usage.
- `access_accounts` — доступы, права, VPN, GitLab, Confluence, 1C.
- `hardware` — ноутбук, принтер, мышь, гарнитура, ремонт оборудования.
- `software_issues` — Chrome, Zoom, антивирус, Excel, Adobe Reader, проблемы после обновления.
- `finance_documents` — счета, акты, закрывающие документы, оплата.
- `hr_requests` — отпуск, больничный, график работы, выход нового сотрудника.
- `document_workflow` — договоры, инструкции, технические задания, согласование.
- `client_partner_requests` — клиенты, партнёры, внешние пользователи, тикеты.
- `meetings_newsletters` — дайджесты, демо, созвоны, плановые работы.
- `spam` — фишинг, подозрительные предложения, срочная верификация.
- `attachments_or_binary` — картинки, бинарные файлы, системные файлы.
- `empty_mail` — пустые письма.
- `needs_review` — письмо не подошло ни под одно правило и требует ручной проверки.

## Как запустить

Из корня проекта:

```bash
bash run.sh
```

Скрипт берёт входящие письма из `data/raw/inbox`, создаёт отдельную папку запуска в `data/processed/`, пишет логи в `logs/` и выполняет базовые проверки.

Можно передать другую папку входящих писем:

```bash
./run.sh /path/to/inbox
```

Можно запустить Python-приложение напрямую:

```bash
python3 -m mail_processor \
  --input data/raw/inbox \
  --output data/processed/manual_run \
  --log-dir logs/manual_run \
  --mode copy
```

Режим `copy` сохраняет исходные файлы. Режим `move` реально перемещает письма:

```bash
python3 -m mail_processor \
  --input data/raw/inbox \
  --output data/processed/moved \
  --log-dir logs/moved \
  --mode move
```

## Как запустить тесты

```bash
python3 -m unittest discover -s tests
```

Тесты проверяют чтение писем, классификацию основных категорий и граничные случаи: пустой файл, бинарный файл, файл без расширения и повреждённый JSON.

## Как устроено решение

1. `MailReader` читает файл и определяет статус чтения: обычный текст, JSON, пустой файл, бинарный файл, изображение, системный файл или ошибка чтения.
2. `RuleBasedClassifier` нормализует текст и считает баллы по ключевым словам. Если ни одно правило не подошло, письмо уходит в `needs_review`.
3. `MailProcessor` применяет чтение и классификацию ко всем файлам папки, создаёт папки категорий, копирует или перемещает файлы.
4. В `processing_log.csv` записывается решение по каждому файлу: категория, статус чтения, score, причина, тема, отправитель и путь назначения.
5. В `stats.json` сохраняется итоговая статистика по категориям и статусам чтения.

## Почему решение устойчивое

В папке входящих есть нестандартные данные: пустой `.txt`, `.bin`, `.jpeg`, `.json` с ошибкой, файл без расширения и `.DS_Store`. Приложение обрабатывает каждый случай явно:

- пустые файлы попадают в `empty_mail`;
- картинки, бинарные и системные файлы попадают в `attachments_or_binary`;
- повреждённый JSON не ломает запуск, а классифицируется по доступному тексту;
- письма без совпадений попадают в `needs_review`, чтобы их разобрал человек и при необходимости добавил новое правило.
